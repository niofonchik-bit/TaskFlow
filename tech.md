# План реализации проекта

## Общие принципы реализации

Проект должен развиваться как современная платформа управления задачами и рабочими процессами, объединяющая основные возможности Trello, Planfix и других подобных систем.

Основной упор необходимо сделать на:

* удобную работу с досками и задачами;
* гибкую организацию команд и рабочих процессов;
* красивый современный интерфейс;
* большое количество уместных анимаций;
* сохранение пользовательских настроек;
* прозрачную историю изменений;
* возможность восстановления архивированных данных;
* демонстрацию качественной архитектуры frontend, API и базы данных.

Проект является демонстрационным пет-проектом. Поэтому не требуется поддерживать миллионы пользователей, огромные объёмы файлов или распределённую инфраструктуру. При этом код, структура базы данных и архитектура должны быть реализованы так, как если бы проект в дальнейшем мог развиваться в полноценный коммерческий продукт.

# Этап 2. Проектирование базы данных

## Общие требования к сущностям

Для основных сущностей необходимо предусмотреть:

* UUID в качестве идентификатора;
* дату создания;
* дату последнего изменения;
* автора создания;
* пользователя, выполнившего последнее изменение;
* дату архивирования;
* пользователя, выполнившего архивирование;
* оптимистическую версию записи или другой механизм защиты от конфликтов.

Архивирование должно быть отделено от окончательного удаления.

Обычный пользовательский интерфейс не должен выполнять физическое удаление данных напрямую.

## Архивирование и восстановление

Задачи:

* [ ] Исключать архивированные записи из обычных запросов.
* [ ] Добавить отдельные endpoints архива.
* [ ] Реализовать восстановление сущностей из архива.
* [ ] Разрешить окончательное удаление только архивированных сущностей.
* [ ] Ограничить окончательное удаление отдельным разрешением.
* [ ] Добавить подтверждение окончательного удаления.
* [ ] Не удалять вложения сразу после обычного архивирования задачи.
* [ ] Удалять физические файлы только при окончательном удалении вложения или задачи.
* [ ] Записывать архивирование, восстановление и окончательное удаление в аудит.

Основные операции:

```text
POST   /api/v1/tasks/:taskId/archive
POST   /api/v1/tasks/:taskId/restore
DELETE /api/v1/tasks/:taskId

POST   /api/v1/boards/:boardId/archive
POST   /api/v1/boards/:boardId/restore
DELETE /api/v1/boards/:boardId
```

`DELETE` должен выполнять окончательное удаление только после архивирования и проверки разрешений.

При архивировании доски её колонки и задачи должны скрываться вместе с доской. Необязательно отдельно изменять `archivedAt` каждой дочерней записи.

## Цветовая система

Цвета не должны храниться в PostgreSQL enum, поскольку набор цветов может расширяться.

Необходимо создать отдельную расширяемую систему цветовых вариантов.

Пример сущности:

```text

├── id
├── key
├── name
├── backgroundColor
├── foregroundColor
├── borderColor
├── isSystem
├── isActive
└── sortOrder
```

Цветовые варианты должны поддерживаться для:

* организаций;
* досок;
* колонок;
* тегов;
* типов задач;
* категорий задач;
* групп пользователей;
* приоритетов;
* пользовательских представлений.

Задачи:

* [ ] Сделать добавление новых цветов возможным без изменения структуры остальных таблиц.
* [ ] Проверять контраст текста и фона.

## Приватность досок

Базовые режимы приватности:

```text
PRIVATE
ORGANIZATION
TEAM
CUSTOM
```

Описание режимов:

* `PRIVATE` — доступ имеют только создатель и явно добавленные пользователи;
* `ORGANIZATION` — доступ имеют все участники организации;
* `TEAM` — доступ имеют выбранные команды;
* `CUSTOM` — доступ определяется выбранными пользователями, командами и группами.

В дальнейшем может быть добавлен режим доступа по публичной ссылке.

# Ограничения пет-проекта

Для сохранения реалистичного объёма разработки не требуется реализовывать:

* хранение файлов большого размера;
* видеоконференции;
* полноценный корпоративный чат;
* сложный BPMN-редактор;
* бесконечно настраиваемый workflow-движок;
* собственный язык автоматизаций;
* микросервисную архитектуру;
* отдельный Elasticsearch;
* сложную обработку документов;
* антивирусную инфраструктуру промышленного уровня;
* полноценную offline-синхронизацию;
* маркетплейс расширений;
* поддержку миллионов одновременных соединений;
* сложную биллинговую систему.

Вместо этого необходимо реализовать небольшое количество функций, но довести их до качественного, законченного и визуально привлекательного состояния.

# Критерии готовности функции

Функция считается завершённой, если:

* создана и применена миграция;
* добавлена серверная валидация;
* реализована проверка прав;
* добавлена бизнес-логика API;
* добавлена обработка ошибок;
* реализован интерфейс;
* реализованы loading, empty и error состояния;
* добавлена адаптация под мобильные устройства;
* добавлены уместные анимации;
* поддержан режим reduced motion;
* добавлены необходимые тесты;
* действие записывается в историю или аудит, если это требуется;
* обновлена OpenAPI-документация;
* функция проверена на локальной и production-конфигурации.
